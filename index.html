📋 Task Management System

Complete Documentation for Students & Trainees | CRUD Application with 3-Tier Architecture

1. Product Overview

1.1 Product Description

A simple task management system implementing CRUD (Create, Read, Update, Delete) operations with status tracking. The application follows a 3-tier architecture pattern with clear separation between presentation, business logic, and data layers.

1.2 Tech Stack

Frontend
Vanilla JavaScript
Backend
Node.js + Express
Database
MySQL 8.0+

1.3 Architecture - 3 Tier System

Tier 1 - Presentation Layer: Browser-based UI (HTML, CSS, Vanilla JS)
Tier 2 - Application Layer: RESTful API (Express.js)
Tier 3 - Data Layer: MySQL Database

2. Database Schema

CREATE TABLE `tasks` (
  `id` int NOT NULL AUTO_INCREMENT,
  `title` varchar(255) NOT NULL,
  `status` enum('ICEBOX','TODO','WIP','DONE') NOT NULL,
  PRIMARY KEY (`id`)
)

Field Descriptions

Field Type Description
id int Auto-incrementing primary key
title varchar(255) Task description (max 255 characters)
status enum ICEBOX (backlog), TODO (ready), WIP (in progress), DONE (completed)

3. Functional Requirements

FR-1: Create Task

  • User can create a new task with a title
  • Default status is TODO
  • Title is required and cannot be empty
  • Maximum title length: 255 characters

FR-2: Read Tasks

  • Display all tasks in a table view
  • Show task ID, title, and current status
  • Visual distinction between different statuses
  • Filter tasks by status

FR-3: Update Task

  • User can edit task title
  • User can change task status via dropdown
  • Validation: title cannot be empty

FR-4: Delete Task

  • User can delete any task
  • Confirmation dialog before deletion
  • Permanent removal from database

Backend: Task Controller (controllers/taskController.js)

const db = require('../config/database');

// Get all tasks
exports.getAllTasks = async (req, res) => {
  try {
    const [tasks] = await db.query('SELECT * FROM tasks');
    res.json(tasks);
  } catch (error) {
    res.status(500).json({ error: error.message });
  }
};

// Create task
exports.createTask = async (req, res) => {
  try {
    const { title, status = 'TODO' } = req.body;
    if (!title) {
      return res.status(400).json({ error: 'Title is required' });
    }
    const [result] = await db.query(
      'INSERT INTO tasks (title, status) VALUES (?, ?)',
      [title, status]
    );
    res.status(201).json({ 
      id: result.insertId, 
      title, 
      status 
    });
  } catch (error) {
    res.status(500).json({ error: error.message });
  }
};

What is 3-tier architecture?
Three-tier architecture separates applications into three logical layers:
  • Presentation Tier (Frontend): User interface - what users see and interact with (HTML, CSS, JavaScript)
  • Application Tier (Backend): Business logic - processes requests, validates data, implements rules (Express.js API)
  • Data Tier (Database): Data storage and retrieval (MySQL)
This separation makes the application easier to maintain, scale, and test.
Why use Vanilla JavaScript instead of React or Angular?
For learning purposes, Vanilla JavaScript is excellent because:
  • You understand how the DOM works without abstractions
  • No build tools or complex setup required
  • Direct manipulation of HTML elements teaches fundamentals
  • Easier to debug and understand what's happening
  • Once you master vanilla JS, frameworks become much easier to learn
What is CRUD?
CRUD stands for the four basic operations of persistent storage:
  • Create: Add new records (POST /api/tasks)
  • Read: Retrieve existing records (GET /api/tasks)
  • Update: Modify existing records (PUT /api/tasks/:id)
  • Delete: Remove records (DELETE /api/tasks/:id)
Almost every application needs CRUD functionality - it's a foundational concept in software development.
How do I handle CORS errors?
CORS (Cross-Origin Resource Sharing) errors occur when frontend and backend run on different ports. Fix by:
  1. Install CORS package: npm install cors
  2. In your Express server.js: 
    const cors = require('cors');
app.use(cors());
This allows your frontend (e.g., port 5500) to communicate with backend (e.g., port 3000).
Why use ENUM for status instead of VARCHAR?
Using ENUM provides several benefits:
  • Data Integrity: Only allows predefined values (ICEBOX, TODO, WIP, DONE)
  • Storage Efficiency: ENUM stores values as integers internally
  • Performance: Faster comparisons and queries
  • Self-Documentation: Schema shows all possible values
  • Prevents Errors: Can't accidentally insert "INPROGRESS" when it should be "WIP"
Should I use callbacks, promises, or async/await?
For modern Node.js development, use async/await:
  • Callbacks: Old style, leads to "callback hell", harder to read
  • Promises: Better than callbacks, but still uses .then() chains
  • Async/Await: Modern, clean syntax, easier error handling with try/catch
// ✅ Use this (async/await)
async function getTasks() {
  try {
    const [rows] = await db.query('SELECT * FROM tasks');
    return rows;
  } catch (error) {
    console.error(error);
  }
}

What's the difference between PUT and PATCH?
  • PUT: Replaces the entire resource - you must send ALL fields
  • PATCH: Partial update - send only fields you want to change
For this simple CRUD app, PUT is fine since we only have 2 fields. For complex resources with many fields, PATCH is more efficient.
How do I prevent SQL injection attacks?
Always use parameterized queries (prepared statements): 
// ❌ DANGEROUS - Never do this
const query = `SELECT * FROM tasks WHERE id = ${id}`;

// ✅ SAFE - Use parameterized queries
const query = 'SELECT * FROM tasks WHERE id = ?';
const [rows] = await db.query(query, [id]);
The ? placeholder is safely replaced by the MySQL driver, preventing malicious SQL injection.
Why do we need a separate models folder?
Separation of concerns is key to maintainable code:
  • Models: Handle database operations (SQL queries)
  • Controllers: Handle business logic and request/response
  • Routes: Define API endpoints and link to controllers
This makes code easier to:
  • Test each component independently
  • Modify database queries without touching business logic
  • Reuse models across different controllers
  • Switch databases (MySQL to PostgreSQL) by only changing models

Common mistakes to avoid?
  • ❌ Not validating user input (always validate on both frontend AND backend)
  • ❌ Hardcoding database credentials (use .env files)
  • ❌ Not handling errors properly (use try/catch blocks)
  • ❌ Forgetting to close database connections (use connection pools)
  • ❌ Not testing API endpoints before building frontend
  • ❌ Mixing concerns (putting SQL queries in routes or controllers)
  • ❌ Not using proper HTTP status codes
  • ❌ Ignoring CORS issues until deployment
